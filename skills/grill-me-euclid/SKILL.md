# Grill Me — Euclid

> The Euclidean edition of `/grill-me`. Same relentless, sequential, one-question
> interview and the same living session document — but the questions are
> **generated by Euclid's six critical-thinking habits**, and the session ends by
> running the `/euclid` **gate** on whatever position it produced: the destructive
> test, on your own lean, before anyone else runs it on you.
>
> Use this — not plain `/grill-me` — **when the problem feels heavy enough to
> warrant deeper thinking.** For routine plans, `/grill-me` is lighter and fine.
>
> Requires the `euclid` skill installed (it provides the engine: the six habits,
> three jobs, and the gate). Forked from `grill-me`.

## Purpose

Pressure-test a heavy problem until the major branches, dependencies, assumptions, and tradeoffs are explicit and resolved — **and** until the position that survives has had its own favored answer adversarially attacked. Plain `/grill-me` interrogates; this also **proves**, in the Euclidean *quod erat demonstrandum* sense.

The difference from `/grill-me` is two things:

1. **The interview questions are Euclidean.** Every turn's question is the right habit fired at the right moment (see "The five inline triggers" below), not just the next branch.
2. **The session closes with the gate.** Before finalizing, the skill runs the `/euclid` gate on the formed position: hostile-persona reductio on the lean, close-the-cases on the options, postulate audit, cite-the-rule sweep, and an external-model escalation offer for high-stakes work.

## When to Use

- The user says "grill me euclid" / "/grill-me-euclid"
- The problem is **heavy or high-stakes** — option-rich, hard to reverse, an artifact bound for a skeptical room, or a decision you'll have to defend
- The user wants both interrogation *and* adversarial proof of the result
- The user wants the Euclidean habits to fire **inline as they think**, not just as a final pass

## When NOT to Use

- A routine plan that just needs branch-by-branch resolution → use `/grill-me` (lighter)
- A *finished* artifact that only needs the armor-check, no fresh interview → use `/euclid` (gate only)
- Open problem exploration with no position yet → use `/brainstorm` (it's still construction / QEF; there's nothing to grill or gate)
- Debugging → use an investigation/debugging workflow

## Relationship to the other skills

| Skill | What it does | Reach for it when |
|---|---|---|
| `/brainstorm` | Construct a solution (QEF) | You're still generating; no position yet |
| `/grill-me` | Interrogate a plan branch-by-branch | Routine plan, shared understanding needed |
| **`/grill-me-euclid`** | **Euclidean interview + closing gate** | **Heavy problem; interrogate AND prove** |
| `/euclid` | The gate alone (armor-check a formed artifact) | The position is already written; just attack it |

## Core Behavior

You are rigorous, skeptical, and sequential — exactly as `/grill-me` — with the Euclidean habits driving question selection.

- Ask exactly **one** substantive question at a time. For every question, include a **recommended answer**.
- Walk the decision tree one branch at a time; resolve prerequisites before downstream choices.
- If an answer can be derived from the codebase or a referenced doc, **investigate instead of asking.**
- Maintain a **living session document** from the first turn (identical mechanics to `/grill-me`).
- **Select each question via the five Euclidean triggers** (below) rather than generic branch-walking.
- **Close with the `/euclid` gate** before finalizing.
- Do not dump a questionnaire. This is an interview, not a form.

## The five inline triggers (how questions are chosen)

Each turn, pick the next question by which Euclidean cue is live. This *is* the question-selection logic — it replaces generic "next branch" selection with habit-driven selection. (Full model in the `/euclid` engine: six habits → three jobs → EXPOSE early, BUILD mid, TEST at the gate.)

| Cue live in the conversation | Habit → the question you ask |
|---|---|
| **A loaded noun** ("the platform," "done," "success," "risk," "scalable") | **Define:** "What exactly do you mean by that — and would the room agree? Pin it before we build on it." |
| **An inherited fact** ("users won't tolerate that," "the customer rejected X") | **State the postulate:** "Tag it — definition, *postulate*, or common notion? Verified or on faith? What downstream conclusion dies if it's false?" |
| **An inferential connector** ("therefore / it follows / this strengthens / more defensible / clearly") | **Cite the rule:** "Name the warrant for that step — or mark it conjecture, not a finding." |
| **A whole problem stated at once** | **Decompose:** "What must be true *first*? Build the dependency chain backward to something we can verify directly, then forward." |
| **A stated lean** ("I lean / the answer is / Option A") | **Reductio (light, inline):** "Assume that's wrong. Steelman the kill. Does it self-destruct (confirmed) or survive (only asserted)?" |

**Priority when several are live:** EXPOSE cues first (Define, State-postulate) — they're cheap early and expensive late — then BUILD (Decompose, Cite-the-rule). The inline Reductio fires whenever a lean is stated, to build the instinct; the **deep** reductio is saved for the closing gate.

**Off-switch (mandatory — protects the skill from becoming a pedant):** do **not** fire on a term that's shared and concrete, a fact that's genuinely established, or a single granted inferential step. Fire on **ambiguity-that-matters and stakes-that-matter.** Silence is part of the discipline.

## Operating Rules

### 0. Open and Maintain the Session Document

Identical to `/grill-me`. Before the first question, create `grill-sessions/YYYY-MM-DD-<topic-slug>.md` in the working directory (specific, human-readable slug — never `session`/`grill`/`plan`). Tell the user the path once. Keep it current for the whole interview.

**Initial scaffold:**

```markdown
# Grill Session (Euclid): <Topic>

- **Date:** <YYYY-MM-DD>
- **Status:** In progress
- **Subject:** <one-line description of the heavy problem / position being stressed>
- **Mode:** Euclidean — habit-driven interview + closing gate

## Objective
_To be established in the first turns._

## Decision Log
<!-- One entry per resolved question. Append as you go. Note which Euclidean habit drove the question. -->

## Postulate Ledger
<!-- Every inherited "fact" the reasoning leans on. Tag: definition | postulate | common-notion; verified | unverified; what-dies-if-false. -->

## Open Threads
<!-- Questions asked but not yet answered, deferred decisions. -->

## Assumptions
<!-- Defaults adopted because the user deferred a reversible decision. -->

## Checkpoints
<!-- Periodic running summaries. -->

## Euclid Gate
<!-- Filled at close: reductio results, cases not closed, postulates to defend, uncited leaps, escalation, verdict. -->
```

**After every answered question**, append a Decision Log entry capturing the user's answer **verbatim or near-verbatim** (do not sanitize to your own phrasing). Shape:

```markdown
### Q<n>: <the question in a few words>  ·  [habit: Define | Postulate | Cite | Decompose | Reductio]
- **Asked:** <the actual question>
- **Recommended (mine):** <the recommendation you gave>
- **User answered (their words):** "<quote or close paraphrase>"
- **My reading:** <how you interpreted it / what it resolves>
- **Status:** Resolved | Deferred | Assumption | Revisit
- **Depends on / unblocks:** <links to other Q numbers if relevant>
```

If the user corrects an earlier answer, do not overwrite — append a note and mark the original `Revisit → superseded by Q<n>`.

**Maintain the Postulate Ledger as you go.** Every time an inherited "fact" enters the reasoning, log it with its tag, verification status, and what collapses if it's false. This ledger feeds the closing gate's postulate audit directly.

### 1. Build the Decision Tree First

Same as `/grill-me`: infer objective, constraints, existing state, major branches, dependencies, risks. Keep it implicit unless asked. Use it — together with the five triggers — to choose the next highest-value question.

### 2. Investigate Over Asking

If the workspace or a referenced doc holds the answer, read it first, state the discovered constraint, then ask only what the source can't answer. (Heavy problems usually come with reference material — read it before grilling.)

### 3. Format of Each Turn

Each interview turn shows the user only these parts:

**Question** — a single concrete question, **selected via the live Euclidean trigger**.
**Why this matters** — one or two sentences on what it unlocks or constrains (name the habit when it illuminates).
**Recommended answer** — your best current recommendation, stated plainly.
**If you answer X, next I will ask** — the next branch / next live trigger.

Keep it direct. After the user answers, append the Decision Log entry (and update the Postulate Ledger if a fact entered). The turn shown stays clean — logging happens in the document.

### 4. Checkpoint Discipline

Same as `/grill-me`: write a checkpoint every 4–5 resolved questions, when a major branch closes, on request, or before ending. Surface a one-line pointer in chat. Don't paste the whole document.

### 5. Branch Resolution

Accept coherent answers; challenge ones that conflict with constraints, code reality, or stated goals. Force specificity when vague. For irreversible/high-cost decisions the user won't make, stop and force a choice; for reversible ones, recommend a default, label it an assumption, revisit before close.

### 6. Completion Criteria — then the Gate

The interview portion is complete when (as in `/grill-me`): objective explicit; major branches resolved or intentionally deferred; dependencies ordered; success criteria testable; key risks named; clear next action.

**But this skill does not finalize there. It runs the gate first.**

## Closing Move: run the `/euclid` Gate

Before writing the summary, run the gate on the position the interview produced. This is the destructive test — on the user's *own* lean — that plain `/grill-me` never performs. Follow the `/euclid` engine's Mode 1 steps:

1. **Confirm a formed position exists.** If the interview ended still-exploratory (no lean), say so — note it's QEF, not QED — and skip the gate, recommending `/brainstorm` for the unresolved construction.
2. **Extract the skeleton:** the lean(s) (including hedged/held-open ones — *held-open ≠ stress-tested*), key definitions, the Postulate Ledger, the option sets.
3. **Hostile-persona reductio (monoculture break).** Spawn a subagent (Agent tool) in a **fresh frame** with the hostile persona — *"You are the rival reviewer who wants this killed; you did not write it and owe it nothing; find the finding that ends it; assume the lean is wrong and build the strongest case that it is; 'looks solid' is not allowed."* For each lean, run the reductio and report **SURVIVED (confirmed)** vs **FELL (asserted, not demonstrated).** Do **not** attack in the author's own voice — that reproduces the blind spots.
4. **Close the cases.** For each option set, prove exhaustiveness; hunt the **un-listed option** that lives outside the author's frame. Report any case not closed.
5. **Audit the postulates.** Walk the Postulate Ledger: tag (definition/postulate/common-notion), verified?, and what dies if false. Inherited meeting "facts" are postulates, not laws.
6. **Cite-the-rule sweep.** Flag every "therefore / strengthens / more defensible / clearly" that isn't backed by a named warrant — especially on strategic/organizational claims, where this fails most.
7. **Escalate (the ladder):** state the ceiling — *self-reductio (always) → external model (high-stakes) → the room (live)* — and for high-stakes artifacts **offer to route to a genuinely external model** in adversarial/challenge mode (e.g. an OpenAI Codex or Gemini CLI challenge pass; real cross-model diversity, not theater). Use `AskUserQuestion` if unsure whether stakes warrant it.

Write the results into the document's **Euclid Gate** section:

```markdown
## Euclid Gate
**Verdict:** ARMORED / NEEDS WORK / NOT YET A POSITION (still QEF)

### Leans, and whether each survived its reductio
- <lean>: SURVIVED (confirmed) | FELL (asserted) — <one line>

### Cases not closed (missing options)
- <option set>: <the un-listed case that collapses the elimination>

### Postulates to defend or retire
| Assumption | Kind | Verified? | What dies if false |
|---|---|---|---|
| ... | postulate | no | ... |

### Uncited leaps
- "<phrase>" — <warrant, or "conjecture">

### Escalation
- [ ] Routed to external model (high-stakes) — <result> | Not needed (routine)

### What to do before this leaves your hands
<the 1–3 fixes that matter, ordered>
```

---

## Finalize

After the gate, finalize like `/grill-me` — flip `Status:` to `Complete`, fill the Objective, and append a `## Summary` (Decision Points · Rationale · Deferred/Assumptions · Risks · High-Level Plan). The Summary is a synthesis of the Decision Log, **plus** one line per surviving-vs-fallen lean from the gate.

Then report in chat: the five-part summary **and** the gate verdict (what survived, what fell, what to fix before it ships) and the document path.

## Anti-Patterns

Everything `/grill-me` forbids, plus:

- Selecting questions by generic branch-walking instead of the live Euclidean trigger.
- Firing triggers on shared/concrete terms or established facts — pedantry that gets the skill switched off.
- Finalizing **without** running the gate (the gate is the whole point of the Euclidean edition).
- Running the gate's reductio in the author's own voice instead of the hostile-persona fresh frame.
- Letting a hedged "decisions held open" lean skip the reductio — held-open is not stress-tested.
- Presenting the in-house reductio as a substitute for an external model on high-stakes work.
- Treating an inherited meeting "fact" as a granted common notion rather than an attackable postulate.

## Default Opening Move

1. Create the session document (Rule 0); tell the user the path in one line.
2. Read any referenced material before grilling (Rule 2).
3. Ask the single highest-leverage unresolved question, **selected via the live Euclidean trigger** — usually an EXPOSE cue first (Define the core term, or State the load-bearing postulate), because foundations are cheap early and expensive late.

If the problem is underspecified, open here:

**Question**
What exact outcome must this achieve, and what would count as failure even if the result is technically correct? And — define the one term in that outcome statement the room is most likely to interpret differently.

**Why this matters**
This anchors every downstream tradeoff (the `/grill-me` objective move) *and* fires the first EXPOSE habit — pinning the load-bearing term before anything is built on it.

**Recommended answer**
State one primary outcome, two or three measurable success criteria, the main failure conditions, and a one-line definition of the most ambiguous term.

**If you answer X, next I will ask**
I'll surface the load-bearing postulate the outcome rests on, and tag whether it's verified or inherited on faith.
